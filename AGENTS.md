# DemoCodex 协作指南

欢迎加入 DemoCodex HTML 小游戏项目！本指南总结了推荐的开发流程与最佳实践，帮助每位贡献者快速融入并保持代码质量。

## 1. 项目结构约定
- `index.html`：页面入口，只保留骨架结构和元信息，复杂逻辑放在脚本中完成。
- `src/css`：样式文件。按页面或组件拆分，公共变量引入后再复用。
- `src/js`：脚本文件。保持模块化，避免将所有逻辑塞进单一文件。
- `assets/`：静态资源。按类型拆分子目录，命名清晰，替换资源时保留原始文件名历史。

## 2. HTML / CSS 最佳实践
- 使用语义化标签（如 `<main>`, `<section>`, `<button>`）提升可访问性。
- CSS 尽量采用 BEM 或同等的命名体系，避免全局样式冲突。
- 优先使用 CSS 变量管理主题色与间距，方便日后换肤或适配深色模式。
- 针对动画和过渡，确保在低性能设备上仍然流畅，可通过 `prefers-reduced-motion` 兼容性处理。

## 3. JavaScript 最佳实践
- 保持函数短小，单一职责，必要时拆分为模块并通过 ES 模块导入。
- DOM 操作集中管理，避免散落在多个模块导致状态难以维护。
- 游戏循环相关逻辑建议封装在自定义对象/类中，便于测试和复用。
- 所有事件监听器在不需要时记得移除，防止内存泄漏。

## 4. 资源与性能
- 尽量压缩图片与音频，优先使用现代格式（WebP/AVIF、AAC/OGG）。
- 运行时大量元素渲染时使用请求动画帧 (`requestAnimationFrame`) 以及节流/防抖策略。
- 对外部依赖保持最小化，能用原生 API 解决的尽量不引入库。

## 5. 测试与浏览器约束
- 浏览器回归脚本只允许使用本机已安装的 `Google Chrome`，统一复用共享的启动与校验逻辑，避免每个脚本各写一套分支。
- 禁止回退到 Playwright bundled browser；不要为本仓库运行 `playwright install`、`playwright install chromium` 或任何下载浏览器的流程。
- 若本机 Chrome 缺失、路径不可执行或被误配到 `ms-playwright` 一类缓存路径，测试脚本应立即失败并给出明确错误，不做静默回退。
- 遵从代码简洁之道：把浏览器路径解析、存在性校验和失败提示收敛到共享 helper，测试场景脚本只保留用例本身。

## 6. 协作流程
- 每次新功能或修复请新建分支，提交信息做到动词开头、描述精准。
- 变更前查看现有 TODO 与 Issue，避免重复劳动。
- 在 PR 中附上演示动图或关键截图，便于评审理解效果。
- 所有逻辑变更需经过本地测试，必要时编写简要的测试步骤说明。

## 7. 可访问性与国际化
- 交互控件需支持键盘操作，必要时设置 `aria-*` 属性。
- 文案统一存放在配置对象中，便于后续多语言拓展。
- 音频反馈需提供静音开关或音量控制。

## 8. 代码审查清单
- 新增或修改的 UI 是否具备响应式与无障碍考虑？
- 是否存在未清理的 `console.log` 与调试代码？
- 动画帧循环是否可控、可暂停？
- 是否更新了相关文档与变更记录？

遵循以上实践，可让 DemoCodex 保持高质量、易维护、更具可玩性。祝开发顺利！
